8) Discussion question: compare methods + memory estimate

Streaming method: easier for live debugging because data shows up immediately, but frequent notifications can bottleneck due to BLE overhead and Python-side processing.
Buffered method: supports faster sampling on the robot (because sampling is local), but requires careful memory management and an extra “dump data” step to transmit the stored values.

How quickly can buffered sampling run? It is mainly limited by the sensor read + array write on the Artemis, not by BLE. BLE is only involved when dumping data afterward.

Memory estimate: The Artemis has 384 kB RAM. If timestamps are stored as 32-bit values (~4 bytes each) and temperatures are 32-bit floats (~4 bytes each), then each paired sample costs ~8 bytes total. In theory, 384,000 / 8 ≈ 48,000 samples could fit if nothing else used RAM, but realistically the program needs stack/other buffers, so the safe number is lower. In my implementation, using arrays of length 1000 only uses ~8000 bytes for the paired data, which is comfortably within RAM.

Lab 2: IMU (ICM-20948) — Pitch/Roll, Filtering, FFT, Complementary Filter

Overview

In this lab, I connected the ICM-20948 IMU to the Artemis using the Qwiic connectors, verified the vendor example, and then built a pipeline to (1) compute pitch/roll from the accelerometer, (2) analyze accelerometer noise using an FFT, (3) apply a low-pass filter to reduce vibration/noise, (4) compute angles by integrating gyroscope rates (and observe drift), and (5) fuse both sensors using a complementary filter for a stable estimate.

Set Up the IMU

Hardware Connections

I connected the ICM-20948 breakout to the Artemis using the Qwiic (I2C) connectors. This provides SDA/SCL plus power/ground with a keyed cable, which is reliable and avoids wiring mistakes.

Example Code + AD0_VAL

I installed the SparkFun 9DOF IMU Breakout_ICM 20948_Arduino Library and ran the Example1_Basics sketch. The AD0_VAL definition controls the least-significant bit of the IMU’s I2C address (it reflects the state of the AD0 pin/jumper). On the SparkFun Qwiic breakout, the default is typically AD0=1 (often corresponding to address 0x69), and closing the ADR jumper sets it to AD0=0 (often address 0x68). The correct value is whichever one matches the hardware jumper configuration.

What I observed when moving the board

When I rotated or flipped the board, the accelerometer axes changed in a way consistent with gravity re-projecting onto the sensor axes. When the board was roughly flat and still, the Z-axis stayed near ~1g (≈1000 mg), while X/Y stayed near ~0 (with noise). The gyroscope measured angular velocity (deg/s), so when the board was not rotating it hovered around 0, and when I rotated it I saw spikes whose sign depended on rotation direction.

Accelerometer: Pitch/Roll from Gravity

Angle Equations

Using the class equations, I computed pitch and roll from accelerometer readings (units can be mg since ratios cancel). A common stable form is: 

// ax, ay, az in mg (or m/s^2), output in degrees
roll_acc  = atan2( ay, az ) * 180/pi
pitch_acc = atan2( -ax, sqrt(ay*ay + az*az) ) * 180/pi

Quiet Data: Sampling + Raw Signals

Data format and 5-second capture

I captured and transmitted a time-stamped IMU log using the format: t_us, ax_mg, ay_mg, az_mg, gx_dps, gy_dps, gz_dps. In my quiet run, I captured 663 samples over ~5.29 s, for an effective sampling frequency of ~125.08 Hz (target IMU period was 5000 µs → ~200 Hz, but overhead reduced the achieved rate).

Raw accelerometer and gyroscope

In quiet conditions, the accelerometer Z-axis stays near ~1g (~1000 mg) with small noise, while X/Y hover near 0. The gyroscope axes hover near 0 deg/s with noise; any bumps/hand motion show up as spikes.

Pitch/Roll from accelerometer (quiet)

The accelerometer-derived pitch/roll estimates are fairly noisy because accelerometers measure both gravity and linear accelerations/vibrations. Even when still, micro-vibrations and sensor noise cause jitter in the computed angles.

Accelerometer Accuracy: Two-Point Calibration

To reduce bias and small scaling error, I applied a simple two-parameter calibration on pitch: 

pitch_calibrated = a * pitch_raw + b

From my calibration run, the fitted parameters were: a = 1.0150104375747704, b = −0.28927677393249907. This slightly increases the scale and shifts the offset so that the “flat” condition is closer to the expected value.

Noise Analysis: FFT and Cutoff Selection

To choose a low-pass cutoff frequency, I computed the FFT of the accelerometer-based pitch signal. The spectrum was broadly noisy rather than dominated by a single sharp tone, meaning much of the jitter is higher-frequency noise rather than slow attitude motion. Since the it changes relatively slowly compared to vibration, a low cutoff is reasonable, hence why I used 5 Hz as a cutoff.

Based on the spectrum and the need to preserve only slow orientation changes, I used a cutoff of fc ≈ 5 Hz for the low-pass filter.

Low-Pass Filter (LPF) on Accelerometer Angles

I implemented a simple first-order IIR low-pass filter: 

// dt in seconds, fc in Hz
alpha = exp(-2*pi*fc*dt)
y[k] = alpha*y[k-1] + (1-alpha)*x[k]

With fc=5 Hz, the filtered pitch angle is noticeably smoother than the raw pitch, while still tracking the slow trend. If fc is too low, the output lags real motion; if it is too high, too much vibration remains.

Gyroscope: Integrated Angles and Drift

I computed pitch/roll/yaw by integrating gyroscope rates over time: 

// omega in deg/s
theta[k] = theta[k-1] + omega[k]*dt

The key issue is drift: even a small constant bias (e.g., 0.2 deg/s) accumulates into degrees of error after a few seconds. In my data, the integrated gyro angles show a clear slow drift despite the board being mostly stationary.

Complementary Filter: Stable + Responsive Pitch/Roll

To combine the short-term accuracy of the gyro with the long-term stability of the accelerometer, I used a complementary filter: 

// gyro_pred = (theta_prev + omega*dt)
// acc_term  = theta_acc_LPF
theta = alpha*gyro_pred + (1-alpha)*acc_term

The results show the expected behavior: the gyro-integrated curve drifts, the accelerometer LPF curve is noisy (especially under impulse/vibration), and the complementary output stays stable without long-term drift while still reacting smoothly.

Vibration Data: Effect on Noise + Filtering

I repeated the same collection/analysis while inducing vibration (placing the IMU on the table and gently striking the table). Compared to quiet conditions, the accelerometer-derived angles become much noisier because vibrations add high-frequency components. This is exactly why the LPF + complementary filter are useful: the LPF suppresses high-frequency accelerometer noise, and the complementary filter prevents gyro drift while rejecting quick accelerometer spikes.

Lab 3: Time-of-Flight (VL53L1X) — Dual Sensors + Non-Blocking Reads + BLE Logging

Overview

In this lab, I integrated two VL53L1X Time-of-Flight (ToF) distance sensors with the Artemis (via the Qwiic I2C bus), and built a data pipeline to (1) run both sensors at the same time, (2) avoid blocking the main loop by checking checkForDataReady(), and (3) record time-stamped distance and time-stamped IMU data for a fixed duration and then transmit the results over Bluetooth (BLE) to my laptop for plotting in Python.

Prelab

I2C address (expected)

The VL53L1X appears on the I2C bus at a default address that I verified by scanning the bus on the Artemis.

Using two sensors: approach + rationale

Because both sensors share the same default I2C address, I needed a method to make them addressable simultaneously. There are two common approaches:

  • Re-address one sensor in software (preferred long-term): boot with only one sensor enabled, change its address programmatically, then enable the second sensor at the default address. After that, both can stay powered and be polled normally. This is convenient because it avoids repeatedly power-cycling sensors.
  • Use XSHUT (shutdown) pins to enable/disable sensors: only one sensor is powered/awake at a time, so both can keep the default address, but you have to toggle shutdown pins whenever switching which sensor is active.

For my robot, I chose to use the XSHUT pins, since I can easily shut or open the pin, thus allowing me to use both ToF sensors simultaneously. This is important for fast sensing.

Planned sensor placement + blind spots

I mounted/plan to mount one ToF sensor facing forward to detect obstacles in the direction of motion, and a second ToF sensor angled to the side to detect walls/side obstacles for future wall-following and obstacle avoidance.

  • Forward sensor: best for frontal collision avoidance and detecting objects directly in the path.
  • Side sensor: best for maintaining distance to a wall and detecting side obstacles during turns.

Scenarios that can be missed: very thin obstacles (small cross-section), obstacles outside the sensor’s field-of-view, obstacles at extreme angles (weak return), and low obstacles that are below the optical path depending on mounting height.

Non-Blocking Sensor Loop + Speed (Limiting Factor)

In future labs, the robot will need fast control loops, so the code cannot block while waiting for a sensor. To enforce this, my loop continuously prints the Artemis clock (or increments a counter) as fast as possible, and only prints new ToF readings when distanceSensor.checkForDataReady() indicates fresh data is available.

How fast does the loop execute?
I benchmarked the loop rate by counting iterations over a fixed time window and computing: loop_hz = iterations / elapsed_time.

Current limiting factor: in practice, the limiting factor is usually the time spent in I2C transactions (sensor reads), plus any serial/BLE printing overhead. If the loop prints too much to Serial, Serial output becomes the bottleneck. My measured loop frequency was around 800 Hz without activating the ToF. With activating the ToF, it shot down to around 160 Hz (with both ToF sensors initialized). As such, it is clear that the loop itself is much faster than the sensor data being ready. Thus, the ToF sensors are clearly the limiting factor in this.

Why separate batteries?

The Artemis and the motors were powered from separate batteries to prevent motor transients (stall current spikes, back-EMF, and supply droop) from brownout-resetting the microcontroller or injecting noise into the logic supply. Even with separate supplies, I tied grounds together so the PWM/logic signals have a common reference.

Parallel-coupling driver channels

To deliver higher current without overheating, I parallel-coupled the two channels on each dual driver so both outputs share the load for one motor. Because both channels are on the same IC (shared timing/clocking), this is a common technique to increase usable current capacity.

A representative gain set for the plotted trials was Kp = 0.12, Ki = 0.05, Kd = 0. Sampling Rate and Extrapolation

A major goal of this lab was to decouple controller rate from sensor rate. In my data, the PID loop ran substantially faster than the ToF sensor. Representative measured rates were about 133–167 Hz for the PID loop and 18.6–19.4 Hz for the ToF sensor, so the controller ran about 6.9× to 9.0× faster than the sensor. slope = (d2 - d1) / (t2 - t1)
d_est = d2 + slope * (now - t2)

The resulting plots show that the extrapolated estimate closely follows the stepwise raw measurement while remaining smooth between ToF updates. This satisfies the lab requirement that the PID loop run faster than the sensor while still using a current distance estimate.

(5000-level) Integrator Wind-up Protection

Because the robot can begin far from the wall, the controller output may saturate for a long time. Without protection, the integral term can continue accumulating during saturation and then cause a large overshoot once the robot gets near the setpoint. To address this, I added an integral clamp and conditional anti-windup logic. I also added a BLE command so wind-up protection can be turned on and off for comparison.

This protection is necessary because the drivetrain is nonlinear: deadband, floor friction, and battery level all change the relationship between PWM and motion. A pure integrator that keeps growing while the actuator is already saturated stores unnecessary error and makes recovery slower and less repeatable. As can be seen, the overshoot is much more severe, with it also taking longer for it to settle. I added 2 videos comparing the windup on and windup off as well.

I did not include a derivative term in the final demonstrated controller because the ToF signal is relatively low-rate and noisy compared to the controller loop, so derivative action was more likely to amplify measurement noise than improve performance. The extrapolator, however, provided a clear benefit because it let the control loop run much faster than the sensor without waiting for new data each cycle.

Yaw Estimate and IMU Considerations

The yaw estimate was obtained by integrating the gyroscope z-axis angular velocity: 

yaw_est_deg = yaw_est_deg + (gz_dps - yaw_bias_dps) * dt

Before each run, I kept the robot still and estimated the gyroscope bias by averaging 300 samples. This was necessary because even a small constant bias would accumulate into heading drift over time when integrated digitally. Since yaw is obtained by integrating gyro rate, any offset in the sensor directly turns into growing angle error. Bias calibration significantly reduced this drift for the time scale of the controller runs.

One thing to note is that taking the derivative of the yaw error would effectively differentiate a signal that already depends on an integrated gyro measurement. In practice, I found that derivative action was not needed here. The IMU already gives angular velocity, and the proportional plus integral terms were enough to achieve the desired turn. Leaving Kd = 0 also avoided derivative kick and reduced sensitivity to noise.

Controller Design

I implemented a PI yaw controller with optional wind-up protection. The controller error is the wrapped difference between the desired heading and the estimated heading: 

err = wrap_deg(yaw_setpoint_deg - yaw_est_deg)
u   = yaw_kp * err + yaw_ki * yaw_i_term + yaw_kd * derr

The wrap function keeps the error in the range of -180 to 180 degrees, which ensures the robot always takes the shortest turning direction. I also used a small stop band near the target. If the error magnitude is below 2 degrees, the PWM command is set to zero so the robot does not chatter around the setpoint.

Because the motors have static friction and a deadband, small controller outputs are not strong enough to make the robot turn. To handle this, I enforced a minimum turn PWM whenever the robot is outside the stop band. In my code, positive turns and negative turns use slightly different minimum PWM values, which compensates for asymmetry in the drivetrain.

I enabled integrator wind-up protection by clamping the accumulated integral term. This is useful because when the robot begins far from the setpoint, the controller output can remain saturated for some time. Without clamping, the integral term would continue growing even though the motors are already at their effective limit, which would produce more overshoot once the robot nears the target.

Lab 7: Kalman Filter

Overview

In this lab, I implemented a Kalman Filter to supplement the slowly sampled ToF distance measurements used in my Lab 5 PID position controller. The Kalman Filter fuses a dynamics model of the robot with noisy sensor readings to produce a faster, smoother estimate of the robot's distance from the wall. This allows the control loop to run at a higher rate than the ToF sensor alone, which is critical for driving toward the wall at higher speeds without crashing.

The lab involved four main steps: (1) performing a step response to estimate the drag and momentum parameters of the robot, (2) building the state space model and initializing the Kalman Filter in Python, (3) sanity-checking the filter on recorded data in Jupyter, and (4) implementing the Kalman Filter on the Artemis and integrating it with my existing PID controller.

Step 2: Initialize the Kalman Filter (Python)

State Space Model

The robot's 1D dynamics are modeled as a second-order system with drag: 

F = u − d·ẋ
m·ẍ = u − d·ẋ

State: x̄ = [x, ẋ]ᵀ  (position and velocity)

A = [[0,    1   ],        B = [[0    ],
     [0,  −d/m  ]]             [1/m  ]]

C = [−1, 0]   (ToF measures negative distance from wall)

I then discretized the matrices using the Euler method with a sampling time of dt ≈ 0.0287 s (approximately 34.9 Hz, the average ToF sample period from my data): 

Ad = I + dt · A
Bd = dt · B
Noise Covariance Matrices

The Kalman Filter requires specifying how much we trust the model versus the sensor. I set the noise covariances as follows:

  • Measurement noise (Σ_z): The ToF sensor has an approximate standard deviation of 20 mm, so Σ_z = (20)² = 400 mm².
  • Process noise (Σ_u): Using the lecture guideline of σ = √(10² / dt) for each state, with dt ≈ 0.0287 s, I obtained σ ≈ 59.1 mm for position and 59.1 mm/s for velocity. This gives Σ_u = diag(3489, 3489).

The relative values of Σ_u and Σ_z determine the filter's behavior. Larger process noise means less trust in the model (the filter follows the sensor more closely), while larger measurement noise means less trust in the sensor (the filter relies more on the model prediction). I discuss the effect of tuning these in the experiments section below.

KF Function

I implemented the Kalman Filter using the function provided in the lab instructions, with variable names following the lecture convention: 

def kf(mu, sigma, u, y):
    # Prediction step
    mu_p = Ad.dot(mu) + Bd.dot(u)
    sigma_p = Ad.dot(sigma.dot(Ad.T)) + Sigma_u

    # Update step
    sigma_m = C.dot(sigma_p.dot(C.T)) + Sigma_z
    kkf_gain = sigma_p.dot(C.T.dot(np.linalg.inv(sigma_m)))

    y_m = y - C.dot(mu_p)
    mu = mu_p + kkf_gain.dot(y_m)
    sigma = (np.eye(2) - kkf_gain.dot(C)).dot(sigma_p)

    return mu, sigma

Effect of Noise Covariance Tuning

To understand how the covariance matrices affect the filter, I ran three experiments:

  • Baseline: The default Σ_u and Σ_z values. The KF closely follows the raw data while smoothing small noise.
  • 10× process noise (less trust in model): The KF hugs the raw sensor data even more tightly, essentially passing through measurement noise.
  • 10× measurement noise (less trust in sensor): The KF relies more heavily on the model prediction and is visibly smoother, but may lag behind rapid changes in the real state.

These experiments confirm the expected behavior from the lecture: the relative balance between Σ_u and Σ_z determines whether the filter favors the model or the sensor. For my application, the baseline parameters worked well because the model is only a rough approximation (the step response did not cleanly reach steady state), so I wanted the filter to still rely significantly on the ToF readings while smoothing the noise between measurements.

Lab 8: Stunts (Task B — Drift)

Overview

In this lab, I combined everything from the previous labs into a single fast stunt. I chose Task B (Drift): the robot starts at the line, drives forward at full speed, and as soon as it gets within about 3 ft of the wall it has to perform a 180° turn and then drive back past the starting line. The stunt is judged on how quickly the robot makes it back across the line, so the goal was to spend as little time as possible braking, turning, and re-accelerating.

My final solution is a three-phase state machine driven by a non-blocking drift_service() on the Artemis: an open-loop full-speed APPROACH, a yaw-PID TURNING phase that reuses my Lab 6 controller, and a full-speed RETURN. The whole thing is started from a single BLE command and dumps time-stamped sensor, motor, and yaw data back to my Jupyter notebook for plotting. The total stunt time for the run shown below was 5.41 s, with the turn itself taking only 1.29 s.

Strategy

The lab instructions explicitly say that the position PID should be disengaged for the approach so the robot does not slow down on its own. My controller from Lab 5 is great at gently parking 1 ft from a wall, but that is exactly the wrong behavior for a stunt where I want the robot to slam toward the wall and only stop when it is time to turn. So my approach was:

  1. APPROACH: drive forward at PWM 255 with no feedback control. Just wait for the front ToF to drop below a tuned threshold.
  2. TURNING: active-brake briefly, then hand off to my Lab 6 yaw PID with a heading setpoint, and wait until the yaw error stays inside a small band for a short hold time.
  3. RETURN: drive forward at full speed (which is now away from the wall) for a fixed duration so the robot crosses the start line.

Each phase logs the raw ToF, the estimated distance, the gyro-integrated yaw, the motor PWM, and the current phase index, so every transition is visible in the post-run plot.

I added a new drift_service() that runs every loop iteration alongside tof_service(), imu_service(), pid_service(), and yaw_service(). It is the only function that owns the motors during a stunt run, and it hands control to the yaw PID only during the TURNING phase. All three phases call drift_log(), which appends a row of (t_us, tof_raw, tof_est, yaw, motor_pwm, phase). After the run, Python pulls those rows in chunks via DRIFT_DUMP_NEXT and turns them into the plots below.

Beating ToF Latency During the Approach

At full PWM the robot covers a lot of ground between ToF samples. The front sensor runs with a 50 ms timing budget, so at roughly 1.5 m/s that is about 75 mm of motion per reading. My first naive version triggered the turn at exactly 914 mm and it crashed into the wall every time, because by the time the controller saw “below threshold” the robot was already much closer. There were three things I had to do to fix this:

  1. Trigger early. Instead of 914 mm, I tuned the threshold up to 1300 mm. The trigger distance and the actual stop distance are not the same number — the trigger has to account for sensor latency plus braking distance. 1300 mm worked reliably, and you can see in the close-up plot that the robot actually crosses 914 mm just before its motors finish reversing.
  2. Hard active brake on entry to TURNING. Inside my motors_stop() I drive all four motor pins HIGH for about 80 ms before releasing. Coast-stopping at 255 PWM was way too slow.
  3. Ignore the first 1 s of ToF. The very first few readings out of tof_service() can be 0 or garbage, which would otherwise instantly trigger the turn at t = 0. I added a drift_tof_settle_ms = 1000 grace period before the threshold check is allowed to fire. You can see this very clearly in the main plot — ToF reads 0 until about t = 1.2 s and then jumps up to ~2200 mm once the sensor stabilizes.

Fixing the Yaw PID for a Real Turn

I had a few problems with the Yaw PID at first. When I first hooked it up to the drift state machine I noticed the robot was essentially doing bang-bang control: it would pin the wheels at the minimum turn PWM all the way through the turn, blow past the setpoint, and then pin them in the opposite direction. The reason was a single block at the bottom of yaw_service(): whenever the PID output dropped below my static-friction floor (185 / 200 PWM), the code would clamp it back up to that floor. So even when the proportional term was telling me to cruise in at PWM 30, the floor was kicking it up to 185. It was to overcome static friction, but I realized it isn't the core essence of what we want from a PID. I did this at first because I wanted it to turn fast, but quickly realized I didn't need it.

I fixed this with a three-zone control law inside yaw_service(): 

if (fabs(err) <= YAW_STOP_BAND_DEG) {
    pwm = 0;                                          // settled
} else if (fabs(err) > 30.0f) {
    // FAR: enforce the static-friction floor so we actually start spinning
    if (err > 0 && pwm > 0 && pwm < MIN_TURN_PWM_POS) pwm = MIN_TURN_PWM_POS;
    if (err < 0 && pwm < 0 && pwm > -MIN_TURN_PWM_NEG) pwm = -MIN_TURN_PWM_NEG;
} else {
    // NEAR: cap the maximum so we don't fly past, with a small floor so wheels still spin
    const int APPROACH_CAP = 130;
    if (pwm >  APPROACH_CAP) pwm =  APPROACH_CAP;
    if (pwm < -APPROACH_CAP) pwm = -APPROACH_CAP;
    const int APPROACH_MIN_POS = 100;
    const int APPROACH_MIN_NEG = 110;
    if (err > 0 && pwm > 0 && pwm < APPROACH_MIN_POS) pwm = APPROACH_MIN_POS;
    if (err < 0 && pwm < 0 && pwm > -APPROACH_MIN_NEG) pwm = -APPROACH_MIN_NEG;
}

I also dropped YAW_MAX_PWM from 255 to 150. Slowing the maximum turn rate gave the IMU integration enough samples to catch the zero crossing instead of slewing past it. It was still more than fast enough as can be seen in my videos below Final yaw gains for the run shown were Kp = 3.3, Ki = 0.08, Kd = 0.20. With the three-zone law in place, the gyro plot in the close-up below shows a clean overshoot to about 138° followed by a smooth pull-back to the setpoint — not the ringing I was getting before.

Final Tuned Parameters

  • Approach PWM: 255 (full speed, open loop)
  • Threshold for trigger: 1300 mm (tuned up from 914 mm to absorb ToF latency + braking distance)
  • Motor calibration factor: 0.75 on the right motor (settable from BLE via SET_MOTOR_CAL)
  • Yaw gains: Kp = 3.3, Ki = 0.08, Kd = 0.20
  • Yaw setpoint (code): 180°
  • Turn settle band: 5° held for 300 ms before exit

Discussion

The most important lesson from this lab was that fast stunts expose every weakness in your sensor pipeline. My Lab 5 PID worked fine for parking 1 ft from a wall because the robot was moving slowly enough that 50 ms of ToF latency was invisible. As soon as I cranked the speed up to PWM 255, that same 50 ms became enough motion to crash the robot before the controller could react. Lab 9: Mapping

Overview

In this lab, I built a 2D map of a static room by placing the robot at five known positions, spinning it in place at each, and logging ToF distance versus IMU-integrated yaw. The scans were transformed into a common room frame using homogeneous transformation matrices and plotted together so I could manually fit straight line segments to the walls and a freestanding obstacle.

I used orientation (PID) control for the turns: 16 commanded increments of 22.5° per scan, with three ToF readings averaged at each stop.

Control Strategy

I reused my Lab 6 yaw PID (Kp = 3.3, Ki = 0.08, Kd = 0.2) and wrapped it in a new non-blocking map_scan_service() that steps through TURNING → SETTLING → SAMPLING sixteen times per 360° scan. Python fires one MAP_SCAN_START BLE command and waits ~60 seconds for completion.

A step settles only when the yaw error is within ±4° and the gyro rate is below 10 °/s, held for 250 ms. The gyro-rate check is essential: without it, the robot would briefly swing through the angle band at speed and trigger a false settle. After settling, SETTLING waits 800 ms for the ToF to stabilize, then SAMPLING averages three fresh readings. 

if (err <= map_settle_band_deg && gz_abs <= 10.0f) {
    map_settle_good_ms += dt_ms;
} else {
    map_settle_good_ms = 0;
}

I also raised the yaw PID's approach-region PWM floor from 100/110 to 120/130. Without that, small 22.5° turns stalled on static friction, registered zero gyro rate, and the settle timer fired on a motionless robot — every step ending ~25° short.

For the mean bias of ~8°, the average placement error is 2 m · sin(8°) ≈ 0.28 m. The radial distance stays correct because it comes directly from the ToF sensor.

Transformation Matrices

Each ToF reading at distance r and measured yaw θ is a point in the sensor frame, then transformed into the room frame by two homogeneous transforms.

Sensor to robot: my front ToF is on the front face of the chassis, pointing along the robot's +x, about dsensor = 5 cm forward of the rotation center. A range r sits at (r + dsensor, 0) in the robot frame: 

T_sensor_to_robot = [[1, 0, d_sensor],
                     [0, 1, 0       ],
                     [0, 0, 1       ]]

Robot to room: a rotation by θ plus translation by the marked position (x0, y0): 

T_robot_to_room = [[cos(θ), -sin(θ), x0],
                   [sin(θ),  cos(θ), y0],
                   [0,        0,       1 ]]

Combined and applied to [r, 0, 1], this simplifies to the closed form I use in Python: 

x_room = x0 + (r + d_sensor) * cos(θ)
y_room = y0 + (r + d_sensor) * sin(θ)

Marked positions ((−3,−2), (0,3), (5,3), (5,−3) ft, plus a (0,0) scan) are converted to meters first.

Lab 10: Grid Localization with a Bayes Filter

Overview

In this lab, I implemented grid localization on the virtual robot using a Bayes filter. The continuous state space is discretized into a 12 × 9 × 18 grid (1944 cells) over x, y, and θ, and the belief is updated every time step with a prediction step that folds in odometry through an odometry motion model and an update step that folds in the 18-beam ToF observation through a Gaussian sensor model.

Prediction Step

The prediction step extracts the actual control (δrot1, δtrans, δrot2) from consecutive odometry readings, then iterates over every (previous state, current state) pair and scores each transition against a Gaussian motion model. To keep the triple loop tractable, I skip any previous state with belief below 10−4 and renormalize at the end. Angles are normalized to [−180, 180)° before they touch the Gaussian — without this, a 350° vs −10° comparison looks catastrophically unlikely even though they represent the same heading. 

for x0, y0, a0 in previous_cells:
    if loc.bel[x0, y0, a0] < 1e-4: continue
    for x1, y1, a1 in current_cells:
        p_trans = odom_motion_model(cur_pose, prev_pose, u_actual)
        loc.bel_bar[x1, y1, a1] += p_trans * loc.bel[x0, y0, a0]

Update Step

The update step is fully vectorized across all 1944 cells. mapper.obs_views has shape (12, 9, 18, 18), and the measurement is broadcast over the last axis to produce a per-beam likelihood cube. Taking a product over the 18 beams gives a single likelihood per cell, which is then multiplied into bel_bar and normalized. Avoiding a Python loop here dropped the update step from seconds to tens of milliseconds. 

per_beam = loc.gaussian(z, mapper.obs_views, loc.sensor_sigma)
loc.bel = loc.bel_bar * np.prod(per_beam, axis=3)
loc.bel /= np.sum(loc.bel)

Results

The plotter screenshot at the top shows ground truth (green), belief (blue), and odometry (red) over the full 16-step trajectory. The belief tracks ground truth tightly around the obstacle loop while odometry drifts off the map entirely — a clean demonstration of why probabilistic localization matters.

Across all 16 steps, the update-step belief index matched the GT index exactly on 8 steps and was within one cell on every other step (position error ≤ 0.305 m, typically much less). Confidences are extremely high after the update (often 1.0), which means the sensor model discriminates well between cells. The prediction step alone is noticeably worse — errors up to 1.05 m, and the bel_bar probability at the true cell occasionally drops to 10−16 or smaller — but the sensor update consistently rescues it. This is exactly the pattern you want: a noisy prior corrected by an informative measurement.

One quirk worth flagging: the printed POS ERROR for yaw grows unboundedly (349° → 706° → … → 6116°) because print_update_stats subtracts the reported GT θ from belief θ without wrapping, and GT θ is itself an unwrapped accumulated angle. Normalizing both into [−180, 180)° gives a small per-step heading error consistent with the 20° grid resolution.

Discussion

The filter works best when the robot is near distinctive geometry — the obstacle corner, the notch, tight walls — where the 18-beam signature uniquely identifies a cell. It is weakest in open symmetric areas and on rotation-dominated steps, where many cells produce nearly identical measurements and the motion model contributes little positional information. The practical takeaway is that odometry is almost useless on its own (the red trail drifts several meters off the true path), but fused with even a noisy ToF observation it becomes a reliable estimator. The vectorized update was also essential: without it, a 1944-cell Python loop inside a 16-step trajectory would have made debugging impossibly slow.

Lab 11: Localization on the Real Robot

Overview

In this lab, I performed grid localization on my physical robot using only the update step of the Bayes filter. The prediction step was omitted because the real robot's motion is too noisy to provide useful odometry information. Instead, I placed the robot at each of four marked positions, performed a full 360° ToF scan with 18 readings at 20° increments, and ran the Bayes filter update to estimate the robot's pose. The key challenge was bridging the gap between the clean simulation environment and the noisy, friction-limited real hardware.

Simulation Baseline

I first verified the provided localization module in simulation using lab11_sim.ipynb. The screenshot below shows the full 16-step trajectory with ground truth (green), belief (blue), and odometry (red). The belief tracks ground truth closely while odometry drifts off the map, confirming the filter works correctly in the ideal case.

Observation Loop Implementation

The real robot's perform_observation_loop() sends a MAP_SCAN_START BLE command with 18 steps. On the Arduino, a non-blocking map_scan_service() state machine drives the motors at full PWM (255) in the correct direction based on gyro-integrated yaw error, coasts when within a 10° settle band, waits for the ToF to stabilize, and logs the measured angle and distance at each step.

An early attempt using PID-controlled turns failed because the robot required near-maximum PWM (~200+) to overcome static friction on the lab floor. With a PID cap of 150, every turn produced the same insufficient torque regardless of error size, resulting in dragging rather than clean rotation. I replaced the PID approach with open-loop bang-bang control: the motors pulse at PWM 255 based on gyro error direction and stop when the gyro indicates the target angle has been reached. This produced reliable turns with a mean angular error of approximately 5° per step.

Because the turns are imprecise, the Python code interpolates the 18 measured (angle, range) pairs onto the exact 0°, 20°, …, 340° grid expected by the Bayes filter using np.interp with wrap-around handling. The BLE notification handler uses byte_array.decode('utf-8', errors='ignore') instead of the default decoder because the Arduino's EString.append(float) occasionally emits non-UTF-8 bytes (0xE5) that would crash the callback and prevent MAP_DONE detection.

I also switched the ToF sensor to long-range mode (TOF_SET_MODE 2) after discovering that the default short-range mode capped readings at approximately 1.3 m, while the map contains corridors up to 3.7 m long. Even in long mode, the ToF became unreliable beyond about 2 m, so I clamped both sensor readings and map expectations to 1.5 m to prevent long-range outliers from dominating the Bayes filter likelihood.

Discussion

The results show a clear pattern: poses surrounded by close walls on most sides localize well, while poses with long open corridors in the scan perform poorly. This is because the VL53L1X ToF sensor becomes unreliable beyond approximately 2 m, returning readings that are significantly shorter than the true distance. When the map expects 3+ meters and the sensor returns 1 meter, the Gaussian likelihood for the correct cell drops dramatically, causing the filter to prefer incorrect cells with coincidentally similar short-range profiles.

Tuning sensor_sigma was critical. In simulation, a tight sigma (0.1) works because the virtual sensor has minimal noise. On the real robot, I needed sigma values of 0.3–1.0 to tolerate the systematic underreading at long range. The range clamping strategy (limiting both sensor data and map expectations to 1.5 m) was the most effective mitigation: it converts long-range beams from penalizing liabilities into neutral observations that neither help nor hurt.

The biggest engineering challenge was the turning mechanism. My robot's motors require near-maximum PWM to overcome static friction on the lab floor, which made the PID controller from Lab 6 (designed for a different surface) completely ineffective. Switching to open-loop bang-bang with gyro feedback was pragmatic: the turns are not precise (5–10° error per step), but the interpolation in Python compensates by mapping measurements onto the correct angular grid using measured rather than commanded angles.
